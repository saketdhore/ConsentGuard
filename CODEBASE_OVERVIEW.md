# ConsentGuard Codebase Overview

ConsentGuard is a Streamlit prototype for health AI compliance triage and patient-facing disclosure generation. The system takes a structured intake, derives reusable legal facts, evaluates declarative law files, and then optionally turns the matched obligations into a deterministic brief and a generated draft document. The project is organized around a clear pipeline: structured intake, fact derivation, rule matching, brief construction, document generation, and document validation.

The main UI lives in [app.py](/C:/ConsentGuard/app.py). It is a single-file Streamlit workflow with views for `landing`, steps `1..11`, `review`, and `results`. `st.session_state.form_data` stores the intake answers and `st.session_state.case_fact_inputs` stores the later case-specific document fields. The UI owns presentation concerns such as labels, step navigation, and preview rendering, but it does not own the legal logic. It submits the standardized taxonomy fields to the backend evaluator, then uses the service layer to build and validate generated documents. The current UI also enforces one taxonomy rule directly: `communication_channel` is only shown when `primary_user == "patient"`.

The evaluation engine is under `engine/`. [engine/engine.py](/C:/ConsentGuard/engine/engine.py) orchestrates the run. It calls [engine/derive_facts.py](/C:/ConsentGuard/engine/derive_facts.py) to enrich raw inputs with reusable legal facts, loads law JSON from [engine/loader.py](/C:/ConsentGuard/engine/loader.py), and uses [engine/matcher.py](/C:/ConsentGuard/engine/matcher.py) to evaluate trigger expressions. The matcher is intentionally small and data-driven: law files express conditions with a compact DSL based on `all_of`, `any_of`, and operators such as `==`, `!=`, `in`, and `contains`. This keeps most rule changes in JSON instead of Python.

The current rule library is centered on Texas. The active Texas files under `laws/TX/HB149` and `laws/TX/HSC183` model four independent Texas checks: commercial biometric privacy, general AI deployment transparency, consumer and healthcare-service disclosure, and licensed practitioner diagnostic use. The engine gates those Texas checks on `jurisdiction == "TX"` and derives helper facts such as healthcare use, emergency-care timing, diagnostic or treatment use, biometric use, and PHI/IIHI handling so the JSON triggers stay readable. The law files return structured obligations, prohibitions, and exceptions rather than raw prose, which is what allows the later brief-builder to remain deterministic.

The schema layer in [engine/schemas](/C:/ConsentGuard/engine/schemas) defines the typed contracts shared across the newer document workflow. `CaseFactsSchema` captures case-specific information such as patient name, provider name, AI system name, and data used. `EvaluationResultSchema` captures the parts of the legal result that the document services need. `ConsentDocumentBrief` is the deterministic bridge between legal matching and language generation. `GeneratedDocumentSchema` describes the structured output expected from OpenAI. `DocumentValidationResultSchema` records whether a draft satisfied the brief, including missing sections, missing points, failed constraints, and warnings.

The service layer in [engine/services](/C:/ConsentGuard/engine/services) turns those schemas into an end-to-end document workflow. [consent_brief_service.py](/C:/ConsentGuard/engine/services/consent_brief_service.py) maps matched obligations into required sections, required points, drafting constraints, timing rules, signature requirements, exceptions, and generation blockers. [document_generation_service.py](/C:/ConsentGuard/engine/services/document_generation_service.py) builds the model input, constructs an OpenAI-compatible structured-output JSON schema, calls the provider, and parses the response into `GeneratedDocumentSchema`. [document_validation_service.py](/C:/ConsentGuard/engine/services/document_validation_service.py) then performs deterministic validation against the brief, checking required sections, required points, consent language, signature requirements, and blockers. If validation fails, the app now still shows the draft for manual review.

OpenAI integration is intentionally thin. [engine/providers/openai_provider.py](/C:/ConsentGuard/engine/providers/openai_provider.py) is a small wrapper around the Responses API and is responsible only for API transport and structured JSON extraction. Environment loading happens in [engine/config.py](/C:/ConsentGuard/engine/config.py), which reads `.env` early in app startup. Users are expected to supply their own `OPENAI_API_KEY`; if the key is missing, the compliance analysis still works, but the document-generation controls are disabled.

From an operational point of view, the repository now has both a rule-evaluation side and a document-generation side. The older smoke-test entry point remains in [test.py](/C:/ConsentGuard/test.py), while automated tests in `tests/` cover the brief builder, document validator, and OpenAI-backed generation service with mocked provider responses. The biggest current constraints are that the Streamlit app is still monolithic, Texas is the only jurisdiction with meaningful populated rule content, and the UI does not yet expose every possible fact needed for every legal pathway. Still, the architecture is coherent and modular: collect structured facts, derive reusable legal meaning, evaluate declarative laws, build a deterministic document brief, generate a structured draft, validate it, and present a usable output to the user.
